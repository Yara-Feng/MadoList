- # 项目名称

  MadoList

  ------

  # 一、项目定位

  ## 产品目标

  帮助关系平等的用户群体共同维护生活计划。

  典型场景：

  - 情侣
  - 室友
  - 同学
  - 家庭成员
  - 学习搭子
  - 健身搭子

  用户可以：

  - 管理自己的待办事项
  - 将部分待办共享给小组成员
  - 查看组员共享待办
  - 提醒组员完成待办
  - 共享查看日程安排

  本产品不是项目管理工具。

  不支持：

  - 任务分配
  - 上下级管理
  - 审批流程
  - 工作流
  - 评论系统
  - 文件附件

  ------

  # 二、业务规则

  ------

  ## 用户规则
  
  每个用户：
  
  ```
必须登录
  只能属于一个小组
  可以不加入小组
  使用微信头像/昵称（需用户授权）
  ```

  ------

  ## 小组规则

  每个小组：

  ```
至少1人
  最多10人
```
  
所有成员：
  
  ```
  地位平等
  无管理员
  无组长
```
  
小组生命周期规则
  
  ```
  当小组最后一个人退出时，小组自动解散.解散后共享待办如何处理？共享待办逻辑删除
创建小组的人退出后，小组仍然存在
  可以重新加入同一个小组
邀请码永不过期，但创建者可手动重置
  ```

  ------

  ## 待办规则
  
  待办分为两种：
  
### PERSONAL
  
个人待办
  
  仅自己可见
  
  ------
  
### GROUP
  
共享待办
  
  组内成员可见
  
  ------
  
  创建后允许切换：

  ```
PERSONAL -> GROUP
  
GROUP -> PERSONAL
  ```

  补充规则：
- **GROUP → PERSONAL 切换后**：已产生的提醒记录和通知保留历史通知，但新用户不可再查看该任务
  - **用户无小组时**：禁止创建 GROUP 类型待办，且禁止将 PERSONAL 切换为 GROUP
- **重复/周期任务：模块8提到了「连续打卡」和重复任务，但待办数据模型中缺少 `isRecurring` / `recurringType` (DAILY/WEEKLY/MONTHLY) 字段。建议在 MVP 阶段至少在数据模型中预留字段。**
  

  
模块4（待办列表）中提到了按「标签」筛选，但：
  - **待办数据模型中没有 `tags` 字段**
- **没有标签管理（创建/删除/编辑标签）**
  
------
  
## 提醒规则
  
用户可以提醒：
  
```
  自己
  组员
```
  
  提醒不会修改任务状态。

  仅产生通知消息。(**微信订阅消息**)

  支持：

  ```
微信授权登录
  ```

  首次登录自动注册用户。

  
  
  微信小程序的登录流程需要注意以下几点，
  - 使用 `wx.login()` 获取 code → 云函数用 code 换取 openid（这是微信云开发的标准流程）
- **头像和昵称获取**：微信从 2022 年 10 月起限制了 `wx.getUserProfile` 接口，建议使用「头像昵称填写」组件（`<button open-type="chooseAvatar">` + `<input type="nickname">`）让用户主动填写
  - 登录时机：`app.js` 的 `onLaunch` 中自动调用云函数登录

  # 模块2 小组系统

  ------

  ## 创建小组

  用户输入：

  ```
  小组名称
  ```

  系统生成：

  ```
6位邀请码
  ```
  
  例如：
  
```
  AB123C
```
  
- 6 位邀请码使用大写字母 + 数字（排除易混淆字符如 0/O、1/I/L），共 30 个字符
  - 邀请码生成时要保证**全局唯一**（云函数中循环检查直到生成不重复的码）
- 需要「复制邀请码」按钮
  
  ------
  
  ## 加入小组
  
  通过邀请码加入。

  限制：

  ```
  用户已有小组
  → 禁止加入
  
小组人数达到10人
  → 禁止加入
```
  
------
  
## 退出小组
  
退出后：
  
```
  保留个人待办
  隐藏共享待办
  ```

# 六、技术要求
  
------
  
  ### 前端

  微信小程序原生

  - 微信开发者工具

  ------

  ### 后端

  微信云函数
  
语言：
  
```
  Node.js
```
  
> **🟢 改进建议 #26 — 后端架构补充 (优先级: 高)**
  
  1. **云函数组织方式**：建议按业务模块组织（action-based dispatch），避免云函数数量过多：
   ```
     cloudfunctions/
   ├── login/          # 微信登录 + 自动注册
     ├── user/           # 用户资料 CRUD
     ├── group/          # 小组：创建/加入/退出/详情/成员列表
   ├── task/           # 待办：CRUD + 状态变更 + 可见性切换
     ├── reminder/       # 提醒：创建提醒 + 防骚扰检查
   └── notification/   # 通知：列表/标记已读
     ```

  2. **每个云函数内部使用 action 分发**：
     ```javascript
     // cloudfunctions/task/index.js
   exports.main = async (event, context) => {
       const { action, data } = event
     switch (action) {
         case 'create': return createTask(data)
       case 'list':   return listTasks(data)
         case 'update': return updateTask(data)
         case 'delete': return deleteTask(data)
       // ...
       }
   }
     ```
  
4. **环境变量**：云函数中通过 `cloud.DYNAMIC_CURRENT_ENV` 使用当前云环境，避免硬编码环境 ID
